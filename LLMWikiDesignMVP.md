# LLMWiki MVP — Phased Design for Business Demonstration

**Version:** 2.0  
**Date:** 2026-05-14  
**Status:** Active — POC Target Selected  
**Goal:** Show the end-to-end value loop to business stakeholders with the minimum viable system, then phase in production complexity after funding.

---

## The One-Line Pitch

> Drop a document anywhere your organization stores knowledge → LLMWiki automatically builds a searchable, interlinked wiki → any AI agent (or human) can ask structured questions and get cited, actionable answers — without ever re-reading the original documents.

---

## Selected POC: UC1 Sales-to-Service Agent via AgentCore

### Why UC1?

| Candidate | Feasibility | Business Visibility | Dependencies | Decision |
|---|---|---|---|---|
| **UC1 Sales-to-Service** | **High** — SOW PDFs available today, no production system writes required | **High** — every engagement starts here, visible to sales and delivery leadership | LLMWiki Phase 1 deployed + /wiki/ask endpoint | **SELECTED** |
| UC2 Environment Provisioning | Medium — needs BOM templates, ARB checklist content | Medium | UC1 must complete first; complex config artifacts | Defer to Phase 3 |
| UC8 Cutover | Low — highest-risk use case, needs full lifecycle context | High | All prior UCs must have contributed; needs HITL governance | Defer to Phase 4 |
| UC10 Hypercare | Low — requires monitoring integration (SageMaker, Arize AI) | Medium | Needs real production data; complex anomaly detection | Defer to Phase 4 |

**UC1 is the right POC target because:**
- Source documents exist today: SOW template, Sales-to-Service deck, customer persona template — all available in AIFactory directory
- No writes to production systems (ServiceNow, QNXT) required — wiki contributions are the only output
- Business outcome is immediately visible: a structured customer handoff brief generated by an agent, sourced from the wiki
- It is the gateway use case — every downstream agent (UC2–UC10) reads what UC1 writes to the wiki
- It validates the full loop: document ingest → wiki page → agent query → structured response → agent contribution — in a single demo

---

## What We Are Proving

The POC demonstrates exactly four things:

1. **Ingest works** — a SOW PDF goes in, structured wiki pages (customer page, scope decisions, source summary) come out automatically via Bedrock Claude
2. **Business API works** — a structured `POST /wiki/ask` call returns a JSON response with `answer + confidence + action_items + artifacts_referenced + gaps_detected`
3. **AgentCore agent works** — the Sales-to-Service Agent deployed in AWS AgentCore calls LLMWiki via registered MCP tools (`wiki_ask`, `wiki_get_customer`, `wiki_get_playbook`, `wiki_contribute`) and produces a customer handoff brief
4. **Knowledge compounds** — the agent writes back a customer context page that the next agent (UC2 Provisioning) can read — proving the handoff mechanism works

## Phase 1 — Business Knowledge API (Weeks 3–4)

### Goal

Add the `/wiki/ask` endpoint that returns structured JSON for agent consumption. This is the unlock for AgentCore integration. Also add the three wiki domain prefixes needed for UC1 (`customers/`, `artifacts/`, `runbooks/`).

### What Phase 1 Adds

| Addition | Why Needed for UC1 POC |
|---|---|
| Business Query Lambda (`lambda/business_query/`) | Returns structured JSON: `answer + confidence + action_items + artifacts_referenced + gaps_detected` |
| `POST /wiki/ask` API route | Primary endpoint for the UC1 AgentCore agent |
| `GET /wiki/playbook/{uc}` route | UC1 agent calls this to get the onboarding playbook |
| `GET /wiki/customer/{id}` route | UC1 agent calls this to check existing customer context |
| `POST /wiki/contribute` route | UC1 agent writes back customer handoff page |
| Contribution Lambda (`lambda/contribute/`) | Validates agent writes, creates S3 page, updates DynamoDB, triggers KB sync |
| S3 wiki domains: `customers/`, `artifacts/`, `runbooks/` | New page types for UC1 content |
| AGENTS.md schema extension | New page types with `use_case_tags`, `domain`, `contributing_agent` frontmatter fields |
| Domain registry (`config/domain-registry.json`) | Maps domain → S3 prefix for `/wiki/query/{domain}` routing |

### Business Query Lambda Design

`lambda/business_query/handler.py` wraps the existing query Lambda with:

1. **Domain routing** — uses `domain` field in request to add KB metadata filter (`use_case_tags contains UC1`)
2. **Intent detection** — classifies intent from question: `retrieve-checklist | retrieve-narrative | retrieve-artifact | retrieve-entity`
3. **Structured response builder** — post-processes Claude's answer to populate `action_items[]`, `artifacts_referenced[]`, `evidence_required[]`, `gaps_detected[]`
4. **Agent context injection** — injects `customer_id` and `project_phase` from request context into the KB retrieval prompt so answers are customer-scoped

### Response Contract (all /wiki/ask responses)

```json
{
  "answer": "synthesized, cited prose",
  "confidence": "high | medium | low",
  "domain": "customer-onboarding",
  "use_case_tags": ["UC1"],
  "sources": [
    {
      "page_slug": "sow-review-checklist",
      "page_type": "runbooks",
      "s3_uri": "s3://llmwiki-bucket/wiki/runbooks/sow-review-checklist.md",
      "relevance_score": 0.94,
      "excerpt": "SOW review step 1: confirm product scope..."
    }
  ],
  "action_items": [
    "Complete persona document using wiki/artifacts/persona-template",
    "Schedule handoff knowledge transfer session"
  ],
  "artifacts_referenced": [
    {"name": "Persona Template", "s3_key": "wiki/artifacts/persona-template.md"}
  ],
  "evidence_required": ["G0 sign-off document", "Customer persona approved"],
  "gaps_detected": [
    {"type": "entity", "slug": "customer-bcbs-mn", "title": "No prior history for BCBS-MN"}
  ],
  "wiki_page_count": 4,
  "contributing_agent_hint": "Contribute customer context page after completing persona"
}
```

### Phase 1 Success Criteria

- `POST /wiki/ask {"question": "What is the UC1 onboarding checklist?", "domain": "customer-onboarding"}` returns structured JSON with `action_items` populated
- `GET /wiki/playbook/UC1` returns step-by-step playbook with at least 3 steps
- `POST /wiki/contribute` creates a new page in `wiki/customers/` and returns `{"status": "indexed"}`
- Bedrock KB re-indexes the contributed page within 5 minutes

### System Prompt (Sales-to-Service Agent)

```
You are the TriZetto Sales-to-Service Onboarding Agent. You are activated when
a customer signs a Statement of Work and transitions from the sales team to the
delivery team.

Your job is to produce a structured customer handoff brief that delivery leads
can act on immediately. You do this by querying the LLMWiki knowledge base using
your tools.

Always:
1. Call wiki_get_customer first to check for prior history.
2. Call wiki_ask with domain "customer-onboarding" to retrieve onboarding best
   practices and delivery risks.
3. Call wiki_get_playbook for UC1 to confirm current onboarding steps.
4. Call wiki_get_artifact for "persona-template" and populate it with data from
   the SOW.
5. Synthesize a Customer Handoff Brief with: customer overview, products in scope,
   key contacts, known delivery risks, action items for the delivery lead, and open
   knowledge gaps.
6. Call wiki_contribute to write the brief back to the wiki.
7. If confidence is "low" for any answer, flag it explicitly in the brief.

Do not make up information. If the wiki does not have an answer, report it as a
knowledge gap — do not invent facts.
```

## Phase 3 — Agent Fleet: All 10 Use Cases (Weeks 8–18)

### Goal

Deploy all 10 use-case agents in AgentCore. Each agent reads from and writes to LLMWiki. Demonstrate the full compounding lifecycle: SOW → provisioning → IAM → config → data migration → SIT → E2E → cutover → handover → hypercare — all knowledge accumulated in LLMWiki.

### Agent Fleet Deployment Order

Agents must be deployed in lifecycle order because each agent reads what the prior agent wrote:

```
Week 8–9:  UC2 Provisioning Agent
Week 10:   UC3 IAM Onboarding Agent
Week 11:   UC4 Business Configuration Agent
Week 12–13:UC5 Data Migration Agent
Week 14:   UC6 SIT Agent
Week 15:   UC7 E2E Testing Agent
Week 16–17:UC8 Cutover Planning Agent
Week 17:   UC9 PTO / Handover Agent
Week 18:   UC10 Hypercare Agent
```

## POC Skill Architecture — UC1 Sales-to-Service Agent

The UC1 POC introduces a **Reusable Skill Registry**: a catalogue of composable agent capabilities that sit between atomic MCP tool calls and the full UC agent logic. Building 5 skills in the POC creates the foundation that every future UC agent inherits.

### Why Skills, Not Monolithic Agent Logic

Without skills, each of the 10 UC agents would re-implement context loading, knowledge querying, gap detection, and artifact retrieval from scratch. With skills, the UC1 POC builds these capabilities once — and UC2 through UC10 wire them at system prompt declaration time with zero Lambda redeployment.

> "The POC pays for itself four times over: 5 skills built for UC1, reused in full or part by all 10 agents."

### The 5 POC Skills

| Skill ID | Business Name | Technical Name | When Called in UC1 | MCP Tool(s) Composed |
|---|---|---|---|---|
| SK-01 | **Customer Briefing Loader** | ContextBootstrapSkill | START of every session — before any other action | `wiki_get_customer` + `wiki_get_playbook` (parallel) |
| SK-02 | **Knowledge Finder** | WikiQuerySkill | Any time the agent needs to look something up | `wiki_ask` (with intent detection + retry on low confidence) |
| SK-03 | **Knowledge Recorder** | WikiContributeSkill | END of session + mid-session for partial contributions | `wiki_contribute` (with human-review routing) |
| SK-04 | **Template Auto-Fill** | ArtifactResolutionSkill | When the agent needs to populate the customer persona template | `wiki_get_artifact` + `wiki_ask` + Bedrock Claude (populate) |
| SK-05 | **Missing Info Radar** | GapDetectionSkill | Whenever SK-02 returns `confidence: "low"` | `wiki_get_gaps` + Bedrock Claude (classify) + SNS (escalate) |

## POC Engagement Harness — Orienting Agents Across Sessions

### Why the Harness Matters for the POC Demo

Without a harness, the UC1 agent is a **stateless function call** — it runs, produces output, and forgets everything. This is fine for a one-time demo but wrong for a real engagement spanning weeks and 10 agents. Anthropic's engineering team documented this failure mode in detail: agents without harnesses either one-shot badly, declare victory too early, or waste time re-discovering state at the start of each session.

The POC adds a lightweight harness on top of the existing skill architecture. The harness uses three artifacts, all stored in the wiki (S3 bucket), to keep every agent session oriented:

| Harness Artifact | Path in Wiki | Purpose |
|---|---|---|
| **Deliverable Registry** | `wiki/engagements/{id}/deliverable-registry.json` | All 12 engagement deliverables, each `"complete": false` until verified |
| **Progress Log** | `wiki/engagements/{id}/progress.json` | What each agent session did — equivalent of `claude-progress.txt` |
| **Health Check** | SK-01 S3 existence checks on prior outputs | Verify prior outputs are actually there before starting new work |

### Harness Flow — What Happens Before UC1 Runs

A new **UC0 Engagement Initializer Lambda** fires on SOW ingest (before UC1). This is the one-time "initializer agent" from the Anthropic pattern:


### Harness Failure Mode Prevention — POC Demonstration

The demo can show two failure modes being prevented:

**1. Premature completion (show in One-Click mode):**

### POC Hard Harness — Phase Implementation Details

This section documents what each phase actually does in the deployed POC, the exact skill invoked, and what gets written to DynamoDB / S3.

#### Phase 1 — SOW Intake & Extraction (`programmatic`)

- **What runs:** Pure Python, no LLM.
- **Action:** Calls `llmwiki-playbook` Lambda with `action=get_customer`. Looks up customer wiki pages.
- **Output fields:** `customer_status`, `pages_found`, `key_facts[]`, `products_in_scope[]`, `overview`
- **Workspace file written:** `phase1-sow-intake.md`
- **Typical latency:** 300–500ms
- **Demo talking point:** *"Phase 1 is pure Python — no AI cost, no hallucination risk. The harness knows what the wiki already knows about this customer before the first LLM call."*

#### Phase 2 — Customer Classification (`llm_single`)

- **What runs:** Single Bedrock Claude call. Schema-validated JSON output.
- **Action:** Classifies customer type, products, risk tier, go-live urgency, implementation complexity.
- **Output fields:** `customer_type`, `products[]`, `risk_tier` (HIGH/MEDIUM/LOW), `go_live_urgency`, `implementation_complexity`, `rationale`
- **Rule:** New customer with no history defaults to `risk_tier=HIGH`.
- **Workspace file written:** `phase2-classification.md`
- **Typical latency:** 2–4s
- **Demo talking point:** *"One Claude call. Structured JSON enforced by regex extraction. The risk tier flows into every downstream phase — Phase 3 questions, Phase 5 risk query, Phase 8 report header."*

#### Phase 3 — Gather Handoff Context (`llm_human_input` — pauses)

- **What runs (first call, `human_context=""`):** Bedrock generates 3 targeted questions informed by Phase 2 classification (risk tier + complexity). DynamoDB status set to `paused`.
- **What runs (resume call, `human_context` filled):** Records context to workspace. Status set to `running`. Phases 4–8 proceed.
- **Async pattern (UI):** UI fires Lambda with `InvocationType="Event"` (returns immediately). Streamlit polls `get_status` every 3 seconds. Phase tiles animate live.
- **Output fields:** `context_provided: true`, `summary` (first 500 chars of human input)
- **Workspace file written:** `phase3-context.md`
- **Demo talking point:** *"The harness pauses after classification — it generates informed questions, not generic ones. It knows this is HIGH risk healthcare before it asks. The pause is a system gate, not a prompt."*

#### Phase 4 — Load Delivery Playbook (`llm_agent` / SK-01)

- **Skill:** `llmwiki-skill-context-bootstrap` (SK-01)
- **Action:** Retrieves UC1 playbook steps and customer-specific pages from wiki.
- **Output fields:** `customer_status`, `playbook_steps`, `pages_loaded`, `skill_latency_ms`
- **Workspace file written:** `phase4-playbook.md`
- **Typical latency:** 300–8000ms (depends on Bedrock KB query)
- **Demo talking point:** *"SK-01 is the context bootstrap. If the wiki has a playbook for this product, it loads it. If not, it records that gap. The harness proceeds either way."*

#### Phase 5 — Risk & Gap Analysis (`llm_agent` / SK-02)

- **Skill:** `llmwiki-skill-wiki-query` (SK-02)
- **Action:** Queries wiki with a product-specific risk question, injecting Phase 3 human context.
- **Output fields:** `confidence` (high/medium/low), `answer`, `action_items[]`, `wiki_page_count`, `skill_latency_ms`
- **Branching rule:** If `confidence=high`, Phase 6 is skipped.
- **Workspace file written:** `phase5-risks.md`
- **Typical latency:** 10–40s (Bedrock KB vector search + Claude synthesis)
- **Demo talking point:** *"The risk query includes the sales context from Phase 3. Same question asked without that context would return generic results."*

#### Phase 6 — Gap Detection & Recording (`llm_batch_agents` / SK-05 ×3)

- **Skill:** `llmwiki-skill-gap-detection` (SK-05) — invoked up to 3 times sequentially (POC; parallel in production)
- **Action:** Derives 3 gap questions from Phase 5 `action_items` + `answer`. Invokes SK-05 once per question. Aggregates all gaps.
- **Skip condition:** `confidence=high` from Phase 5 → skip entirely, write `skipped=true`.
- **Output fields:** `gaps[]` (each with `gap_type`, `title`, `blocking`, `human_prompt`, `gap_id`), `gap_count`, `sub_agents_run`, `blocking_count`
- **Workspace file written:** `phase6-gaps.md`
- **Typical latency:** 10–30s for 3 sub-agents
- **Demo talking point:** *"Three sub-agents, one per gap question. Each runs the same SK-05 skill with a different focused question. Blocking gaps surface as 🚨 in the panel — these must be resolved before go-live."*

#### Phase 7 — Template Population (`llm_agent` / SK-04)

- **Skill:** `llmwiki-skill-artifact-resolution` (SK-04)
- **Action:** Passes an `available_context` bundle (all phases 1–6) and asks SK-04 to locate and fill the `persona-template` artifact.
- **Output fields:** `found`, `completion_pct`, `populated_fields[]`, `missing_fields[]`, `artifact_content`, `skill_latency_ms`
- **Workspace file written:** `phase7-persona.md`
- **Typical latency:** 100–500ms (template lookup)
- **Demo talking point:** *"SK-04 knows what was gathered across all prior phases. What it can't fill is as valuable as what it fills — the missing fields become action items for the services team."*

#### Phase 8 — Write Handoff + Report (`llm_single` / SK-03)

- **Skill:** `llmwiki-skill-wiki-contribute` (SK-03)
- **Action (SK-03):** Indexes the full handoff brief as `wiki/customers/{customer_id}-harness-handoff-{year}.md`.
- **Action (S3):** Writes a structured plain-text report to `wiki/reports/{customer_id}-handoff-report-{date}.txt`. Generates 1-hour presigned URL.
- **Output fields:** `indexed`, `s3_uri`, `report_s3_key`, `report_download_url`, `report_uploaded`, `skill_latency_ms`
- **Workspace file written:** `phase8-handoff.md`
- **DynamoDB update:** `status=completed`, `total_latency_ms`, `completed_at`
- **Typical latency:** 1–3s
- **Demo talking point:** *"SK-03 writes to the wiki so future harness runs for this customer find their own history in Phase 1. The system improves with each run. Download button appears — this is a real deliverable."*

### Async Polling Architecture (implemented)

```
UI: user submits Phase 3 answer
    │
    ▼
lambda.invoke(InvocationType="Event")   ← returns in <100ms
    │                                      (fire and forget)
    ▼
uc1_harness runs phases 4–8 in background
    │  After each phase:
    │  DynamoDB.update(current_phase=N, phase_results=...)
    │
UI: polls get_status every 3 seconds
    │  get_status = DynamoDB.get_item → returns phases_done + phase_results
    │
    ▼
8 phase tiles animate live:
  ✅ Phase 1   ✅ Phase 2   ✅ Phase 3
  🔵 Phase 4   ⬜ Phase 5   ⬜ Phase 6   ⬜ Phase 7   ⬜ Phase 8
    │
    ▼ (when status=completed)
Full phase_results fetched → completion message + download button
```

**Key properties:**
- User sees progress within 3 seconds of submitting Phase 3 answer
- No Streamlit timeout risk — UI is only doing lightweight DynamoDB reads
- If browser closes during polling, state is safe in DynamoDB — re-open page and poll again
- `get_status` responds in ~100ms from DynamoDB (no Lambda cold start cost on poll path)

### Known Limitations (POC → Production gaps)

| Item | POC behaviour | Production fix |
|---|---|---|
| DOCX output | Plain-text `.txt` uploaded to S3 | Lambda container image with `python-docx` layer |
| Phase 6 sub-agents | Sequential (3 × SK-05 in a loop) | Parallel via `asyncio.gather` or Step Functions Map state |
| Phase 3 resume | Polls for paused run by `engagement_id` query — picks latest | Lock by `run_id` passed from UI |
| Report presigned URL | Expires in 1 hour | Increase TTL or use CloudFront signed URL |
| Harness timeout | 900s Lambda hard limit | Split phases 5–8 into a second Lambda or use Step Functions |
| Multi-user isolation | Single engagement_id per customer (latest run wins) | Add `agent_session_id` as secondary partition |
